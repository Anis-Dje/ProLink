# Pro-Link

**Pro-Link** is a Flutter-based professional management application developed
for **Université Constantine 2 – Abdelhamid Mehri**, Department of Fundamental
Computing and its Applications (IFA), as part of the **2025-2026 Mobile
Development Project**. It bridges the gap between the university and the
corporate world by streamlining the entire internship lifecycle.

## Overview

Pro-Link unifies three user roles under a single credential system:

| Role | Who | What they do |
| --- | --- | --- |
| **Admin** | HR / University Coordinators | Approve interns, assign mentors & departments, publish schedules / policies, manage users |
| **Mentor** | Professional supervisors / Teachers | Evaluate interns, take weekly attendance, upload training material |
| **Intern** | Students | View digital Work-ID, schedules, training files, evaluations |

## Features

### Authentication
- Unified login screen for all three roles
- Self-service registration for interns (admin approval required)
- Mentor / Admin accounts created from the admin console
- Password reset via email

### Admin
- Pending intern approval queue (approve / reject with reason)
- Assign interns to mentors and departments
- Upload office schedules (PDF / image) with weekly labels
- Upload policy handbooks
- Enable / disable any account

### Mentor
- Dashboard with assigned-intern summary
- Rate interns on six criteria (0-20) with auto-computed overall score
- Weekly attendance matrix (present / absent / late / justified)
- Upload training resources with searchable tags

### Intern
- **Digital Work-ID** with photo, student ID, department, status badge and QR code
- Weekly schedule viewer
- Training catalog with predictive search
- Evaluation history with per-criterion breakdowns and running average

### Core
- Predictive search (schedules, training, interns, policies)
- File uploads for images, documents, PDFs
- Role-based routing via `go_router`
- Navy-blue corporate theme (#1a2332) across every screen

## Tech Stack

- **Framework**: Flutter 3+ (Dart 3+)
- **State Management**: [`provider`](https://pub.dev/packages/provider) +
  a small `ChangeNotifier` (`AuthStateNotifier`) driving the router
- **Navigation**: [`go_router`](https://pub.dev/packages/go_router) with
  role-aware redirects
- **Backend**: Firebase (Auth + Firestore + Storage)
- **Additional**: `cached_network_image`, `file_picker`, `image_picker`,
  `qr_flutter`, `url_launcher`, `shared_preferences`

## Getting started

### 1. Install Flutter
Follow the [official Flutter install guide](https://docs.flutter.dev/get-started/install)
for your platform, then verify:

```bash
flutter --version
flutter doctor
```

### 2. Get dependencies
```bash
flutter pub get
```

### 3. Configure Firebase
Install the FlutterFire CLI (one-time) and then run `flutterfire configure`
pointed at your own Firebase project. This will generate a real
`lib/firebase_options.dart` — overwriting the placeholder shipped in this
repo — plus `android/app/google-services.json` and
`ios/Runner/GoogleService-Info.plist`.

```bash
dart pub global activate flutterfire_cli
flutterfire configure --project=<your-firebase-project-id>
```

Your Firebase project needs the following services enabled:

| Service | Used for |
| --- | --- |
| **Authentication → Email/Password** | Login, registration, password reset |
| **Cloud Firestore** | All app data (users, interns, evaluations, ...) |
| **Cloud Storage** | Profile photos, schedules, training documents |

### 4. Seed an admin account
On a fresh Firebase project no admins exist yet. Create one either by running
the app, registering an intern, and then manually flipping that user's `role`
to `admin` in the Firestore console, or by calling
`AuthService.createMentorOrAdmin` from a trusted environment.

### 5. Run
```bash
flutter run
```

## Firestore collections

The app uses the following top-level collections:

- `users` — every authenticated identity (admin, mentor, intern)
- `interns` — extended profile for intern-role users
- `departments`
- `mentors`
- `evaluations`
- `attendance`
- `schedules`
- `training_files`
- `notifications`

Recommended security rules are role-aware — admins read/write everything,
mentors read/write only documents owned by themselves or their assigned
interns, interns read/write only their own documents. Starter rules can be
derived from the `UserRole` enum in `lib/models/user_model.dart`.

## Development notes

- Lints are defined in `analysis_options.yaml` and based on
  `flutter_lints`.
- The design language is a dark navy theme built around `#1a2332` — do
  not reach for `Colors.white` / `Colors.black` directly; use
  `AppColors` instead.
- All user-facing strings are in French to match the target audience
  (Université Constantine 2).
- Firestore model (de)serialization lives inside each model's
  `fromFirestore` / `toFirestore` factory — avoid leaking
  `cloud_firestore` types further up the stack.
